# 02-06 User

## Purpose

The User is the **human actor entity** in Framee. Every person who interacts with the system — whether as an employee, manager, administrator, or system process — is represented as a User record. The User entity stores authentication credentials, profile information, and role assignments.

The User system is the final layer connecting the abstract concepts of roles and permissions to real people who log in, create records, approve documents, and interact with the ERP system every day.

---

## Goals

1. Provide secure user account management with authentication credentials.
2. Link users to one or more roles that define their permissions across the system.
3. Support profile information relevant to ERP context (name, email, department, position).
4. Enable user-level settings such as language preference, timezone, and notification preferences.
5. Support user lifecycle management: active, inactive, invited, locked.
6. Enable multi-tenant user isolation — users are scoped to a single tenant.

---

## Scope

### In Scope
- User account creation and management (CRUD)
- Password management (creation, change, reset via email)
- Role assignment (one or more roles per user)
- User profile fields (name, email, phone, avatar, department)
- User status management (active, inactive, locked)
- User invitation flow (invite by email)
- Login session tracking
- User preference settings (language, timezone, date format)
- System Manager user management capabilities

### Out of Scope
- OAuth / SSO authentication (future enhancement)
- Multi-tenant user (a user belongs to exactly one tenant)
- User groups (future — group-based access beyond roles)
- API key management for service accounts (future)

---

## Functional Requirements

### FR-001 User Registration and Invitation
- Users are created by System Managers — self-registration is disabled by default.
- An invited user receives an email with a setup link to set their password.
- The invitation link expires after a configurable duration (default: 48 hours).

### FR-002 Authentication
- Users authenticate with `email` + `password`.
- Passwords are hashed using bcrypt before storage. Plain text passwords are never stored.
- Failed login attempts are tracked. After 5 consecutive failures, the account is temporarily locked for 30 minutes.

### FR-003 Role Assignment
- A user can be assigned one or more roles via the User form's Roles tab.
- Role assignment changes take effect immediately (permission cache is invalidated).
- A user with no roles assigned cannot log in (or can log in with Guest permissions only, depending on configuration).

### FR-004 User Status
- `active` — Can log in and use the system.
- `inactive` — Account disabled, cannot log in, sessions revoked.
- `invited` — Invitation sent, password not yet set.
- `locked` — Temporarily locked due to failed login attempts.

### FR-005 Auto-Lock & Session Security
- **Idle Timeout**: If a user is inactive (no mouse movement, scrolling, or typing) for a configured period (e.g. 15 minutes), the frontend session automatically locks.
- **PIN Unlock**: A locked session overlays the screen with a PIN entry component. The user must enter their 6-digit `pin_hash` to unlock.
- **Failed PIN Limits**: If the PIN is entered incorrectly 3 times consecutively, the session is forcibly terminated (Logout), and the user is redirected to the main Login screen.

### FR-005 User Profile
- Name (full name), email, phone, avatar image, department, job title.
- Email is unique per tenant — it is the login identifier.

### FR-006 User Preferences
- Language (e.g., `en`, `id`) for UI localization.
- Timezone for datetime display.
- Date format preference.
- Notification preferences (in-app, email — managed by Notification plugin).

### FR-007 System User
- A built-in `Administrator` user exists per tenant with System Manager role.
- This user is created during tenant provisioning and cannot be deleted.

## Architecture

```
sys_user
  │
  ├──► sys_user_role (FK: user_id + role_id) → Permission Engine
  │
  ├──► JWT token issuance on login
  │       { user_id, tenant_id, roles[], session_id }
  │
  └──► All CRUD operations carry created_by / updated_by = user.id
```

### Authentication Flow

```
1. User submits email + password to POST /api/v1/auth/login
2. API Engine finds user by email + tenant_id
3. Verify password hash (bcrypt.compare)
4. Check user status = active
5. Load user's roles
6. Issue JWT access token (15min) + refresh token (7d)
7. Store refresh token in Redis (for revocation)
8. Return tokens + user profile
```

| Column | Type | Description |
|--------|------|-------------|
| `id` | VARCHAR(36) | UUID PK |
| `tenant_id` | VARCHAR(36) | Tenant reference |
| `user_id` | VARCHAR(36) | FK → sys_user.id |
| `role_id` | VARCHAR(36) | FK → sys_role.id |

| `assigned_at` | DATETIME | Role assignment timestamp |

### Indexes

```sql
-- sys_user
UNIQUE INDEX idx_user_email (tenant_id, email)
INDEX idx_user_status (tenant_id, status)
INDEX idx_user_tenant_deleted (tenant_id, is_deleted)

-- sys_user_role
UNIQUE INDEX idx_user_role_unique (tenant_id, user_id, role_id)
INDEX idx_user_role_user (tenant_id, user_id)
INDEX idx_user_role_role (tenant_id, role_id)
```

### Redis Keys

| Key | TTL | Description |
|-----|-----|-------------|
| `framee:session:{session_id}` | 900s | Active session data |
| `framee:refresh:{token_hash}` | 7d | Refresh token (revocation check) |
| `framee:login_fail:{email}` | 30m | Failed login counter |
| `framee:perm:{tenant_id}:{user_id}` | 300s | Compiled permission set |

## API Design

### Authentication Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/api/v1/auth/login` | Authenticate with email + password |
| `POST` | `/api/v1/auth/logout` | Revoke current session |
| `POST` | `/api/v1/auth/refresh` | Refresh access token |
| `POST` | `/api/v1/auth/change-password` | Change own password |
| `POST` | `/api/v1/auth/forgot-password` | Request password reset email |
| `POST` | `/api/v1/auth/reset-password` | Reset password with token |
| `GET` | `/api/v1/auth/me` | Get current user profile + roles |
| `PUT` | `/api/v1/auth/me` | Update own profile and preferences |

### User Management Endpoints (System Manager Only)

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/v1/doc/User` | List all users |
| `POST` | `/api/v1/doc/User` | Create / invite a user |
| `GET` | `/api/v1/doc/User/:id` | Get user details |
| `PUT` | `/api/v1/doc/User/:id` | Update user |
| `DELETE` | `/api/v1/doc/User/:id` | Soft delete (deactivate) user |
| `POST` | `/api/v1/doc/User/:id/activate` | Activate user |
| `POST` | `/api/v1/doc/User/:id/deactivate` | Deactivate user |
| `POST` | `/api/v1/doc/User/:id/unlock` | Unlock locked user |
| `POST` | `/api/v1/doc/User/:id/roles` | Update role assignments |

#### Example — `POST /api/v1/auth/login`

**Request:**
```json
{
  "email": "user@company.com",
  "password": "SecurePassword123"
}
```

**Response:**
```json
{
  "success": true,
  "data": {
    "access_token": "eyJ...",
    "refresh_token": "eyJ...",
    "expires_in": 900,
    "user": {
      "id": "user-uuid",
      "full_name": "Budi Santoso",
      "email": "user@company.com",
      "roles": ["Sales Manager", "All"],
      "language": "id",
      "timezone": "Asia/Jakarta"
    }
  }
}
```

## UI Behaviour

### Login Page
- Email and password fields.
- "Forgot Password" link.
- Error messages: "Invalid credentials", "Account locked", "Account inactive".
- Redirect to the dashboard after successful login.

### User Profile Page (`/app/profile`)
- Edit own full_name, phone, avatar, language, timezone, date_format.
- Change password section (requires current password).

### User List View (Admin)
- Columns: Name, Email, Status badge, Roles, Last Login.
- Filter by status, role.
- Bulk actions: Activate, Deactivate.
- "Invite User" button opens invitation form.

### User Form View (Admin)
- **General section**: Full Name, Email, Phone, Department, Job Title, Avatar.
- **Status section**: Status, Lock information.
- **Roles tab**: Multiselect list of roles to assign.
- **Preferences tab**: Language, Timezone, Date Format.
- **Activity tab**: Last login, IP address, session history.

## Validation Rules

- `email` must be a valid email format and unique per tenant.
- `password` must meet all configured complexity requirements.
- `full_name` is required.
- Invited users cannot be assigned roles until they complete account setup (optional — configurable).
- A System Manager cannot deactivate their own account.
- Soft-deleting a user revokes all active sessions and refresh tokens.
- `invitation_token` is one-time use — it is cleared after the user sets their password.

---

## Security

### Password Security
- Passwords are hashed with bcrypt (cost factor: 12).
- Plain text passwords are never logged or stored.
- Password reset tokens are cryptographically random (32 bytes) and expire within 1 hour.
- Old passwords cannot be reused (last 3 passwords are hashed and checked).

### Session Security
- Access tokens expire after 15 minutes (configurable, max 60 minutes).
- Refresh tokens are rotated on every use — a refresh invalidates the previous refresh token.
- All active sessions are invalidated when a user's status changes to inactive/locked.
- Logout explicitly revokes the refresh token in Redis.

### Audit
- All login attempts (success and failure) are logged with timestamp and IP.
- Role changes are logged with the assigning user.
- Password changes and resets are logged.

## Performance

- User authentication is optimized with a direct index lookup on `(tenant_id, email)`.
- Compiled permission sets are cached in Redis per user (TTL: 300 seconds).
- User profile is included in the login response — no separate profile API call needed on login.
- Session data in Redis has short TTL (15 minutes) to minimize Redis memory footprint.

## Acceptance Criteria

- [ ] `POST /api/v1/auth/login` with valid credentials returns an access token and refresh token.
- [ ] `POST /api/v1/auth/login` with wrong password returns 401 and increments the failed login counter.
- [ ] After 5 consecutive failed logins, the account is locked and returns 423 Locked.
- [ ] A locked account can be unlocked by a System Manager via the admin endpoint.
- [ ] `POST /api/v1/auth/refresh` exchanges a valid refresh token for a new access token and rotates the refresh token.
- [ ] Using a refresh token twice returns 401 (token already rotated).
- [ ] `POST /api/v1/auth/logout` revokes the refresh token and subsequent refresh attempts return 401.
- [ ] Deactivating a user revokes all their active sessions immediately.
- [ ] `GET /api/v1/auth/me` returns the current user's profile and role list.
- [ ] Assigning a role to a user updates their permissions within the cache TTL window.
- [ ] A user without any roles receives permissions only from the `All` role.
- [ ] `email` uniqueness is enforced — a duplicate email within the same tenant returns 422.
- [ ] Passwords are stored as bcrypt hashes — no plain text in the database.
- [ ] The invitation link expires after the configured `USER_INVITATION_EXPIRY_HOURS`.

---

## Notes

- Users in Framee are always tenant-scoped. There is no concept of a global user who spans multiple tenants. If the same person needs access to multiple tenants, they must have separate user accounts in each tenant.
- The `Administrator` system user is created during tenant setup. It has the `System Manager` role and cannot be deleted. Its password must be changed immediately after first login.
- User email addresses serve as both the login identifier and the contact point for system notifications. Changing a user's email requires re-verification to prevent account hijacking.
- AI agents generating user or role configurations should always default to the principle of least privilege — create roles with minimum necessary permissions rather than copying System Manager permissions.
