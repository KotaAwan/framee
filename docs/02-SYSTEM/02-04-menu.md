# 02-04 Menu

## Purpose

The Menu system defines the **navigation structure** of the Framee application. It controls what users see in the sidebar, how navigation items are organized, and which DocTypes or pages each menu item links to.

Rather than hard-coding navigation in the frontend, Framee's navigation is fully metadata-driven. Menu items are stored as records that reference Modules and DocTypes — making navigation configurable, permission-aware, and extendable by plugins without frontend code changes.

---

## Goals

1. Provide a flexible, metadata-driven navigation system for the entire application.
2. Render navigation dynamically based on active modules, user role permissions, and menu configuration.
3. Allow administrators to customize navigation order, grouping, and labels without code changes.
4. Support plugin-contributed menu items that integrate seamlessly with the core navigation.
5. Ensure that users only see menu items for DocTypes they have permission to access.

---

## Scope

### In Scope
- Menu item definition (DocType links, module groups, external links, dividers)
- Menu item ordering and grouping within modules
- Role-based menu item visibility
- Module-level menu section rendering
- Plugin-contributed menu items
- Top-level sidebar navigation generation API

### Out of Scope
- Breadcrumb rendering (handled by the frontend layout component)
- Page routing logic (handled by Next.js App Router)
- Permission enforcement on DocType access (handled by Permission Engine)
- Mobile bottom navigation bar (frontend UI concern)

---

## Functional Requirements

### FR-001 Menu Item Types
- `module_group` — A collapsible section header grouping menu items under a module.
- `doctype_link` — A link to a DocType's list view.
- `custom_link` — A link to a custom page URL (internal or external).
- `divider` — A visual separator between menu items.

### FR-002 Module Group
- Module groups appear as top-level sections in the sidebar.
- They are auto-generated from active modules but can be customized in order and label.
- Each module group contains its associated DocType links.

### FR-003 DocType Link
- A menu item of type `doctype_link` generates a navigation entry to `/app/{module}/{doctype}`.
- DocType links are automatically hidden if the user has no `read` permission on the DocType.

### FR-004 Custom Links
- A custom link item can point to any internal or external URL.
- Custom links can be restricted to specific roles.

### FR-005 Ordering
- Menu items within a module have a configurable `sort_order`.
- Module groups themselves have a `sort_order` (inherited from `sys_module.sort_order` by default, overridable in the menu).

### FR-006 Role-Based Visibility
- Each menu item can have an `allowed_roles` list.
- If `allowed_roles` is empty, the item is visible to all authenticated users (subject to DocType permission for doctype_link items).
- If `allowed_roles` is set, only users with at least one matching role see the item.

### FR-007 Plugin Menu Items
- Plugins declare their menu items in `plugin.manifest.json`.
- The Plugin Loader creates menu records on plugin registration.

---

## Architecture

```
Navigation Render Flow:

1. Frontend requests /api/v1/meta/navigation (on login)
2. API Engine serves the navigation manifest for the current user
3. Navigation manifest = active modules + menu items filtered by:
   a. is_active = true
   b. User has read permission on DocType (for doctype_link items)
   c. User's role is in allowed_roles (if set)
4. Frontend caches the navigation manifest in Zustand store
5. Sidebar renders from the cached manifest
```

## Database Design

### `sys_menu` Table

| Column | Type | Description |
|--------|------|-------------|
| `id` | VARCHAR(36) | UUID PK |
| `tenant_id` | VARCHAR(36) | Tenant reference |
| `module_id` | VARCHAR(36) | FK → sys_module.id (parent module group) |
| `label` | VARCHAR(150) | Display label |
| `type` | VARCHAR(30) | module_group / doctype_link / custom_link / divider |
| `doctype` | VARCHAR(100) | DocType name (for doctype_link type) |
| `url` | VARCHAR(500) | URL (for custom_link type) |
| `icon` | VARCHAR(100) | Icon name or SVG identifier |
| `sort_order` | INT | Display order within module |
| `is_active` | TINYINT(1) | Enable/disable this item |
| `allowed_roles` | JSON | Array of role names (empty = all authenticated) |
| `open_in_new_tab` | TINYINT(1) | Open link in new browser tab |
| `plugin_name` | VARCHAR(100) | Registering plugin (null for manual) |
| `parent_id` | VARCHAR(36) | FK → sys_menu.id (for nested menus, future) |
| `created_by` | VARCHAR(36) | FK → sys_user.id |
| `updated_by` | VARCHAR(36) | FK → sys_user.id |
| `created_at` | DATETIME | Auto timestamp |
| `updated_at` | DATETIME | Auto timestamp |
| `is_deleted` | TINYINT(1) | Soft delete |

### Indexes

```sql
INDEX idx_menu_module (tenant_id, module_id, sort_order)
INDEX idx_menu_tenant_active (tenant_id, is_active)
INDEX idx_menu_doctype (tenant_id, doctype)
INDEX idx_menu_type (tenant_id, type)
```

## API Design

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/v1/meta/navigation` | Get navigation manifest for current user |
| `GET` | `/api/v1/doc/Menu` | List all menu items (admin) |
| `POST` | `/api/v1/doc/Menu` | Create a menu item |

| `DELETE` | `/api/v1/doc/Menu/:id` | Soft delete menu item |

## UI Behaviour

### Sidebar Navigation
- The sidebar renders module groups as collapsible sections.
- Active module groups expand by default; others are collapsed.
- The current page's menu item is highlighted.
- Sidebar can be collapsed to icon-only mode on smaller screens (mobile-first).

### Menu Item Management (Admin)
- Admin users access the Menu list from System Settings.
- Menu items can be reordered via drag-and-drop.
- Adding a new item opens a form with type-specific fields (DocType selector for doctype_link, URL input for custom_link).
- Module group items are automatically generated from active modules and cannot be deleted (only reordered).

### Permission-Aware Rendering
- DocType link items are server-side filtered by DocType read permission.
- Custom link items are filtered by `allowed_roles`.
- A module group with no visible items is not rendered (collapsed or hidden).

## Configuration

| Config Key | Default | Description |
|------------|---------|-------------|
| `MENU_CACHE_TTL` | `300` | Navigation manifest cache TTL (seconds) |
| `MENU_AUTO_GENERATE_FROM_MODULE` | `true` | Auto-create doctype_link items for new DocTypes |

---

## Validation Rules

- `type` must be one of: `module_group`, `doctype_link`, `custom_link`, `divider`.
- `doctype` is required when `type = doctype_link` and must reference a valid, active DocType.
- `url` is required when `type = custom_link`.
- `label` is required for all types except `divider`.
- `sort_order` must be a positive integer.
- `allowed_roles` must be an array of valid role names (empty array means no restriction).

---

## Security

- `/api/v1/meta/navigation` is permission-aware — the server filters items based on the requesting user's roles and permissions.
- Admin menu management endpoints require System Manager or Module Manager role.
- Custom link URLs are validated to prevent open redirect attacks — external URLs are flagged in the admin UI.
- `allowed_roles` is enforced server-side, not just on the frontend.

## Events

| Event | Trigger |
|-------|---------|
| `menu.navigation_requested` | User requests navigation manifest |
| `Menu.after_insert` | New menu item created |
| `Menu.after_update` | Menu item updated |
| `Menu.after_delete` | Menu item soft deleted |

### Listened Events

| Event | Action |
|-------|--------|
| `module.activated` | Auto-generate doctype_link menu items for new module |
| `module.deactivated` | Deactivate all menu items for the module |
| `DocType.activated` | Auto-create doctype_link menu item if `MENU_AUTO_GENERATE_FROM_MODULE = true` |
| `plugin.registered` | Register plugin-declared menu items |

---

## Performance

- Navigation manifest is cached per `(tenant_id, user_id)` in Redis with a 300-second TTL.
- Cache is invalidated when menu items change, user roles change, or DocType permissions change.
- The navigation manifest is fetched once on login and stored in the frontend Zustand store.
- Subsequent navigations within the app use the cached manifest — no repeated API calls.

## Future Improvements

- **Nested Menus** — Support sub-menus (two levels) for very large module navigation trees.
- **Favorites** — Users can pin frequently used DocType links as favorites.
- **Quick Access** — A "Recent" section showing the last 5 DocTypes the user accessed.
- **Custom Dashboards** — Module groups can link to custom dashboard pages instead of list views.
- **Search in Navigation** — Global command palette for quick navigation to any DocType or record.

## Acceptance Criteria

- [ ] `GET /api/v1/meta/navigation` returns only module groups for active modules.
- [ ] `GET /api/v1/meta/navigation` excludes doctype_link items for DocTypes the user cannot read.
- [ ] `GET /api/v1/meta/navigation` excludes custom_link items where the user's role is not in `allowed_roles`.
- [ ] A new DocType with `MENU_AUTO_GENERATE_FROM_MODULE = true` gets a menu item automatically.
- [ ] Reordering menu items updates `sort_order` and the sidebar reflects the new order.
- [ ] A plugin's declared menu items appear in the navigation after plugin registration.
- [ ] A deactivated module's menu group disappears from the navigation.
- [ ] Navigation manifest is served from Redis cache on repeat requests (cache hit confirmed via event log).
- [ ] A divider type item renders as a visual separator in the sidebar.

---

## Notes

- Navigation is served as a manifest that the frontend renders — the server does not render navigation HTML.
- Module groups in the navigation correspond directly to `sys_module` records. The label and icon on the menu can override the module's defaults.
- For large enterprises with many modules (20+), consider implementing a search-in-navigation feature to allow quick access without scrolling through the sidebar.
- The navigation manifest must be treated as security-sensitive — leaking menu item names can reveal which business modules are active, which is information competitors should not have.
