# 02-04 Menu & Workspace Architecture

## Purpose

The Menu and Workspace system defines the **navigation structure** and **role-based access** of the Framee application. It controls what users see in the sidebar based on their assigned roles.

Rather than hard-coding navigation in the frontend, Framee's navigation is fully metadata-driven based on the following hierarchy:

`Module << Menu << Workspace << Role`

---

## Goals

1. Provide a flexible, highly normalized, metadata-driven navigation system.
2. Render navigation dynamically based on a user's role mapping via the Workspace table.
3. Automatically inherit display properties (Module Name, Module Icon, DocType Icon) through table relationships rather than duplicating data.
4. Ensure users only see menu items for DocTypes they are explicitly granted access to via their Role-Workspace mapping.

---

## Architecture & Concept

The core concept relies on strict hierarchical relationships:

1. **Role (`sys_role`)**: The user's role (e.g., System Administrator).
2. **Workspace (`sys_workspace`)**: A mapping table that assigns a `menu_id` to a `role_id`. This defines the specific order (`sort_order`) that a role sees a menu item.
3. **Menu (`sys_menu`)**: The actual navigation item. It strictly references a `doctype`.
4. **DocType (`sys_doctype`)**: Defines the target resource and holds its own `icon`.
5. **Module (`sys_module`)**: The DocType belongs to a Module. The Module provides the `name` and `icon` for the top-level grouping in the Sidebar.

```
Sidebar Rendering Flow:

1. Get current User's Roles.
2. Query `sys_workspace` for all records matching the User's `role_id`s, ordered by `sort_order`.
3. Join `sys_workspace.menu_id` to `sys_menu`.
4. Join `sys_menu.doctype` to `sys_doctype` (to get doctype `icon` and `module_id`).
5. Join `sys_doctype.module_id` to `sys_module` (to get module `name` and `icon`).
6. Group the results by Module for sidebar rendering.
```

## Database Design

### `sys_menu` Table

Defines the available navigation items in the system.

| Column | Type | Description |
|--------|------|-------------|
| `id` | VARCHAR(36) | UUID PK |
| `code` | VARCHAR(50) | Unique Code (e.g. MENU-YYYY-XXXX) |
| `name` | VARCHAR(150) | Display name in the menu (e.g., "User") |
| `doctype` | VARCHAR(100) | FK → sys_doctype.table_name |
| `is_deleted` | TINYINT(1) | Soft delete |
| `status` | VARCHAR(50) | Active / Draft |

*(Also includes standard log and version tables: `sys_menu_logs`, `sys_menu_version`)*

### `sys_workspace` Table

Acts as the intersection/mapping table between Roles and Menus.

| Column | Type | Description |
|--------|------|-------------|
| `id` | VARCHAR(36) | UUID PK |
| `code` | VARCHAR(50) | Unique Code (e.g. WORK-YYYY-XXXX) |
| `name` | VARCHAR(150) | Label/Identifier for this mapping record |
| `role_id` | INT | FK → sys_role.id |
| `menu_id` | INT | FK → sys_menu.id |
| `sort_order`| INT | Display order for this menu item within the role |
| `is_deleted` | TINYINT(1) | Soft delete |
| `status` | VARCHAR(50) | Active / Draft |

*(Also includes standard log and version tables: `sys_workspace_logs`, `sys_workspace_version`)*

## UI Rendering Behavior

### Sidebar Data Mapping
When rendering the Sidebar, the frontend relies on the following relationship chains provided by the API:

- **Module Group Header**: Uses `module.name` and `module.icon`.
- **Menu Item Label**: Uses `menu.name`.
- **Menu Item Icon**: Uses `doctype.icon`.
- **Target URL**: The route structure uses the standard format `/[module]/[doctype]` as implemented in the frontend router.

### Visibility
If a Menu item is not mapped to any of the User's roles via the `sys_workspace` table, the User will not see it. If an entire Module has no accessible Menu items, the Module group is hidden completely.

---

## Configuration & Routing

- All standard menu items will target the default List View page for their respective DocTypes: `/[module]/[doctype]`.
- Non-doctype routes (like custom dashboards or external links) are handled via custom logic or extensions to this base architecture if required in the future.
