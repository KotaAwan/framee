# 02-05 Role

## Purpose

The Role system defines the **access control groups** in Framee. A Role is a named set of permissions that can be assigned to one or more users. Instead of managing permissions per-user (which is unscalable in an ERP system with hundreds of employees), permissions are managed per-role and users are assigned to roles.

Roles are the primary mechanism through which the Permission Engine determines what any given user can see and do in the system.

---

## Goals

1. Provide a named, reusable permission grouping that can be assigned to multiple users.
2. Support role hierarchy — child roles inherit all permissions of their parent roles.
3. Enable fine-grained permission assignment per DocType and per field to each role.
4. Allow system administrators to define, modify, and audit role assignments.
5. Support reserved system roles that cannot be deleted or have core permissions removed.

---

## Scope

### In Scope
- Role definition management (name, label, description, parent role, system flag)
- Role hierarchy with permission inheritance
- DocType-level permission assignment to roles (via the Permission Engine integration)
- Role listing and search for user assignment
- System-reserved roles: `System Manager`, `Guest`, `All`

### Out of Scope
- User-to-role assignment (managed in the User DocType — see `02-06`)
- Field-level permission configuration (managed via `sys_docfield_permission` — see `01-07`)
- Permission enforcement logic (handled by the Permission Engine — see `01-07`)

---

## Functional Requirements

### FR-001 Role Definition
- A role must have a unique `name` per tenant.
- `name` is the programmatic identifier; `label` is the display name.
- Roles can have an optional `description` explaining their purpose.

### FR-002 System Roles
- The following roles are pre-created and cannot be deleted:
  - `System Manager` — Full access to all system features and all DocTypes.
  - `All` — A virtual role that represents all authenticated users. Permissions on `All` apply to every user.
  - `Guest` — An unauthenticated user. Only public endpoints apply to Guest.

### FR-003 Role Hierarchy
- A role can declare a `parent_role_id` pointing to another role.
- A child role inherits all permissions of its parent.
- Inheritance is additive — a child always has at least as many permissions as the parent.
- The hierarchy tree must not contain cycles.

### FR-004 Permission Configuration on Role
- From the Role form, an administrator can configure DocType-level permissions for the role.
- This directly creates/updates records in `sys_permission`.
- Supported actions per DocType: `read`, `write`, `create`, `delete`, `submit`, `cancel`, `export`, `share`.

### FR-005 Role Search for Assignment
- A searchable list of roles is available when assigning roles to users.
- Role search returns `name`, `label`, and `description`.

## Architecture

```
sys_role
  │
  ├──► sys_user_role (many-to-many with sys_user)
  │
  └──► sys_permission (one-to-many: role → DocType permissions)
        │
        └──► sys_docfield_permission (one-to-many: permission → field permissions)
```

## Database Design

### `sys_role` Table

| Column | Type | Description |
|--------|------|-------------|
| `id` | VARCHAR(36) | UUID PK |
| `tenant_id` | VARCHAR(36) | Tenant reference |
| `name` | VARCHAR(100) | Unique role name per tenant |
| `label` | VARCHAR(150) | Display label |
| `description` | TEXT | Role purpose description |
| `parent_role_id` | VARCHAR(36) | FK → sys_role.id (hierarchy) |
| `is_system_role` | TINYINT(1) | Reserved, cannot be deleted |
| `created_by` | VARCHAR(36) | FK → sys_user.id |
| `updated_by` | VARCHAR(36) | FK → sys_user.id |
| `created_at` | DATETIME | Auto timestamp |
| `updated_at` | DATETIME | Auto timestamp |
| `is_deleted` | TINYINT(1) | Soft delete |

### Indexes

```sql
UNIQUE INDEX idx_role_name (tenant_id, name)
INDEX idx_role_parent (parent_role_id)
INDEX idx_role_system (tenant_id, is_system_role)
```

### Relationships

```
sys_role.id ← sys_user_role.role_id      (many-to-many via sys_user_role)
sys_role.id ← sys_permission.role_id     (one-to-many)
sys_role.parent_role_id → sys_role.id    (self-referential hierarchy)
```

## UI Behaviour

### Role List View
- Shows all roles with name, label, parent role, and user count.
- Filter by system roles, parent role.
- Quick action: "Assign Users".

### Role Form View
- **General section**: Name, Label, Description, Parent Role.
- **Permissions tab**: A table of all DocTypes with checkboxes for each permission action.
  - Rows: DocType names (one per row)
  - Columns: read, write, create, delete, submit, cancel, export, share
  - Checkbox state reflects inherited permissions from parent role (dimmed checkboxes = inherited)
- **Users tab**: Read-only list of users assigned this role.

### Permission Inheritance Display
- Inherited permissions from parent role are shown as greyed-out checked boxes.
- Explicit permissions (set on this role directly) are shown as checked and editable boxes.
- Removing an explicit permission reverts to the inherited value (if any).

## Configuration

| Config Key | Default | Description |
|------------|---------|-------------|
| `ROLE_MAX_HIERARCHY_DEPTH` | `5` | Maximum depth of role hierarchy |
| `SYSTEM_ROLES` | `['System Manager', 'All', 'Guest']` | Reserved system role names |

---

## Validation Rules

- `name` must be unique per tenant. Duplicates are rejected.
- `name` must match `^[A-Za-z][A-Za-z0-9 _-]*$`.
- System roles (`is_system_role = 1`) cannot be deleted.
- `parent_role_id` must not create a circular dependency (a role cannot be its own ancestor).
- Role hierarchy depth must not exceed `ROLE_MAX_HIERARCHY_DEPTH`.
- System role permissions for `System Manager` cannot be restricted.

---

## Security

- Role management requires System Manager role.
- System roles are protected — their core permissions cannot be removed.
- Role permission changes immediately invalidate the permission cache for all users in that role.
- Roles are tenant-scoped — a role in one tenant does not affect another tenant.

## Events

| Event | Trigger |
|-------|---------|
| `Role.after_insert` | New role created |
| `Role.after_update` | Role modified |
| `Role.after_delete` | Role soft deleted |
| `role.permission_changed` | A DocType permission on this role changed |

### Listened Events

| Event | Action |
|-------|--------|
| `role.permission_changed` | Invalidate permission cache for all users with this role |
| `Role.after_delete` | Remove all sys_user_role records referencing this role |

---

## Performance

- Role list (names and IDs) is cached for link field autocomplete lookups.
- Role hierarchy is pre-flattened during permission cache compilation — no recursive query at runtime.
- Permission cache for a user is refreshed when their role assignment changes (event-driven invalidation).

---

## Notes

- The `All` role is a special virtual role. Any permission assigned to `All` applies to every authenticated user in the tenant. Use with caution — typically `All` should have minimal or no write permissions.
- The `Guest` role applies to unauthenticated requests. In most ERP deployments, `Guest` has no permissions at all.
- When designing role structures, the principle of least privilege applies: start with minimal permissions and expand as needed.
